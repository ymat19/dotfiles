# Prompt Review

Claude Code のプロンプト（CLAUDE.md, AGENT.md, SKILL.md 等）をコンテキストエンジニアリングの原則で評価・改善する。

核心原則: **最小のトークンで最大の期待結果尤度を最大化する。**

## ワークフロー: 評価モード

### Phase 1: 対象特定

1. ユーザー指定のファイルがあればそれを対象とする
2. 指定がなければ以下を自動検出:
   - プロジェクトの `CLAUDE.md`, `AGENT.md`
   - `~/.claude/CLAUDE.md`（グローバル設定）
   - 引数で渡された SKILL.md
3. 対象ファイルを Read ツールで読み取る

### Phase 2: 評価

各ファイルを6軸で評価し、スコアリングテンプレート（後述）で結果を出力する。

- 良い点も明示する（改善だけでなく維持すべき点も）
- 問題には具体的な該当箇所を引用する

### Phase 3: リファクタリング提案

- 優先度 [高][中][低] で改善点を整理
- 具体的な変更案を提示（抽象的な助言ではなく実際のテキスト）
- **ユーザー承認なしに変更を適用しない**

### Phase 4: 適用

ユーザー承認後:
- Edit ツールで変更を適用
- 変更前後の差分を表示

## ワークフロー: 作成モード

プロンプト（CLAUDE.md, AGENT.md, SKILL.md 等）を新規作成・編集する際のワークフロー。

**絶対原則: Claude の自己判断によるコンテクスト生成を許可しない。** すべての記述は "Effective Context Engineering for AI Agents" (Anthropic Applied AI Team) の原則に基づくこと。

### Step 1: 目的の明確化

- 対象読者（人間 / エージェント / 両方）を特定
- このプロンプトが解決する具体的な問題を1文で定義
- Just-In-Time で取得可能な情報を前もって含めない

### Step 2: 構造設計

- マークダウンヘッダーで明確なセクション分割
- 各セクションが独立して意味を成すよう設計（→ Compaction Resilience）
- 概要 → 詳細 → 例 → 注意事項の論理的順序

### Step 3: コンテンツ執筆

Signal Density と Right Altitude を意識して書く:

- 各文のチェック: 「この文を削除してプロンプトの効果が変わるか？」→ 変わらなければ削除
- 具体的なヒューリスティクスを提供（「適切に処理する」は禁止）
- 条件分岐の過剰指定を避ける（脆い手順書にしない）
- 例を効果的に使う — 例は千の言葉に値する

### Step 4: Context Budget 最適化

- 常にロードされるべき情報 vs 参照で十分な情報を分離
- 軽量識別子（パス、コマンド名）で参照し、詳細はツールで取得する設計
- 巨大なコードブロックや JSON 定義をインライン展開しない

### Step 5: 自己レビュー

- 後述の評価基準6軸で自己採点
- 各軸 4/5 以上を達成するまで改善
- 特に Signal Density と Actionability を重視

### アンチパターン集

以下は記事の原則に反するパターン。検出したら修正すること:

| アンチパターン | 違反する原則 | 修正方法 |
|-------------|------------|---------|
| 「適切に対応してください」 | Altitude（曖昧） | 具体的な判断基準を示す |
| 同じ指示を異なる言い回しで繰り返す | Signal Density | 1回だけ明確に述べる |
| 設定ファイル全体をインライン記載 | Context Budget | パスで参照し必要時に読み取る設計に |
| セクション間の暗黙の依存 | Compaction Resilience | 各セクションに必要な文脈を自己完結させる |
| 原則の列挙のみで例がない | Actionability | 具体例を追加 |
| ハードコードされた条件分岐の連鎖 | Altitude（脆い） | ヒューリスティクスに書き換え |

## 評価基準: 6軸

### 1. Altitude（抽象度） — 適切な高度で飛んでいるか

行動を導くのに十分具体的、かつヒューリスティクスとして柔軟であること。

**スコアリング:**

| スコア | 基準 |
|--------|------|
| 5 | 判断のヒューリスティクスを与えつつ、未知の状況にも対応できる柔軟性がある |
| 4 | 概ね適切だが、一部が具体的すぎるか抽象的すぎる |
| 3 | 機能するが、脆い条件分岐か曖昧な指針が混在 |
| 2 | 大部分がハードコードされたロジック、または抽象的すぎて行動指針にならない |
| 1 | 完全に脆い手順書、または意味のない一般論のみ |

**チェックポイント:**
- 過剰な条件分岐（if X then Y else Z の連鎖）がないか
- 「適切に対応してください」のような空虚な指示がないか
- 判断基準が示されているか（何をもって良い/悪いとするか）

**例:**
- 悪い（脆い）: `ファイルが100行以上なら分割し、関数が3つ以上あればモジュール化せよ`
- 悪い（曖昧）: `コードを適切に整理してください`
- 良い: `単一責任の原則に従い、1ファイルが複数の独立した責務を持つ場合は分割を検討せよ`

### 2. Signal Density（信号密度） — すべてのトークンが場所を正当化しているか

> "Find the smallest set of high-signal tokens that maximize the likelihood of your desired outcome."

**スコアリング:**

| スコア | 基準 |
|--------|------|
| 5 | 削除可能なトークンがほぼない。すべてが行動変容に寄与 |
| 4 | 軽微な冗長性はあるが、全体として密度が高い |
| 3 | 一部に冗長な説明や繰り返しがある |
| 2 | 多くの文が自明か、同じことの言い換え |
| 1 | 大部分が情報を持たないフィラーテキスト |

**チェックポイント:**
- 同じ指示の繰り返しがないか
- 自明な説明（「これは重要です」等の装飾）がないか
- 「ファイルを読んで」→「Read ツールで」のように具体的・簡潔にできないか
- 1文削除してもプロンプトの効果が変わらないなら、その文は不要

### 3. Structure & Navigation（構造と導線） — スキャン可能で参照しやすいか

**スコアリング:**

| スコア | 基準 |
|--------|------|
| 5 | 明確なヘッダー階層、論理的順序、テーブルやリストの活用。即座に目的のセクションに到達可能 |
| 4 | 概ね良い構造だが、一部のセクション配置が最適でない |
| 3 | ヘッダーはあるが、情報の論理的グルーピングが不十分 |
| 2 | フラットな構造で、関連情報が散在 |
| 1 | 構造なし。散文の壁 |

**チェックポイント:**
- マークダウンヘッダーで明確にセクション分割されているか
- 概要→詳細→例→注意事項の論理的順序か
- 重要な情報が長い段落の中に埋もれていないか
- テーブル、リスト、コードブロックが適切に使われているか

### 4. Context Budget Awareness（コンテキスト予算意識） — 有限資源として扱っているか

コンテキストウィンドウは有限。トークンが増えるほど注意が分散し、想起精度が低下する（Context Rot）。

**スコアリング:**

| スコア | 基準 |
|--------|------|
| 5 | 必要最小限の情報のみ含み、詳細は参照先に委譲。Just-In-Time 取得パターンを活用 |
| 4 | 概ね効率的だが、一部にインライン展開すべきでない詳細がある |
| 3 | 情報量は適切だが、構造化による圧縮の余地がある |
| 2 | 大量の詳細がフロントロードされ、注意を分散させる |
| 1 | コンテキスト予算を考慮していない。全情報をダンプ |

**チェックポイント:**
- 軽量識別子（パス、リンク）で参照し、必要時にデータを取得する設計か
- 段階的開示（Progressive Disclosure）のパターンがあるか
- 常に必要な情報 vs 必要時のみ参照する情報が分離されているか
- CLAUDE.md に巨大なコードブロックや設定をインラインで埋め込んでいないか

### 5. Compaction Resilience（圧縮耐性） — 圧縮後も重要情報が生存するか

長い会話でコンテキスト圧縮が発生した際、重要な指示が失われないか。

**スコアリング:**

| スコア | 基準 |
|--------|------|
| 5 | 各セクションが独立して意味を成す。重要情報がヘッダー直下に構造化されている |
| 4 | 概ね独立しているが、一部のセクション間に暗黙の依存がある |
| 3 | 重要情報は存在するが、散文に埋もれている箇所がある |
| 2 | セクション間の依存が強く、部分的な読み取りで意味が失われる |
| 1 | 全体を読まないと個々の指示が理解できない |

**チェックポイント:**
- 各セクションが独立して意味を成すか
- 重要な決定事項が明示的に構造化されているか（散文に埋もれていないか）
- アーキテクチャ決定、規約、制約がヘッダー付きで明記されているか
- 冗長な散文の中に重要な1文が隠れていないか

### 6. Actionability（実行可能性） — エージェントが何をすべきか明確か

> "For an LLM, examples are the 'pictures' worth a thousand words."

**スコアリング:**

| スコア | 基準 |
|--------|------|
| 5 | 具体的なヒューリスティクス、コマンド例、出力例がある。読んだ直後に行動可能 |
| 4 | 概ね行動可能だが、一部に曖昧な指示がある |
| 3 | 何をすべきかは分かるが、具体的な方法が不明確な箇所がある |
| 2 | 原則は述べているが、具体的な行動に変換しにくい |
| 1 | 抽象的な原則や理念のみ。行動指針がない |

**チェックポイント:**
- 具体的なコマンド例・コードスニペットがあるか
- Few-shot の例が効果的に使われているか
- 「適切に」「必要に応じて」等の曖昧語が多用されていないか
- 成功条件・完了条件が定義されているか

## 出力テンプレート

評価結果は以下の形式で出力する:

```markdown
## 評価結果: <ファイル名>

| 基準 | スコア | 概要 |
|------|--------|------|
| Altitude | X/5 | 一行の所見 |
| Signal Density | X/5 | 一行の所見 |
| Structure | X/5 | 一行の所見 |
| Context Budget | X/5 | 一行の所見 |
| Compaction Resilience | X/5 | 一行の所見 |
| Actionability | X/5 | 一行の所見 |
| **総合** | **X/5** | 総合評価 |

### 維持すべき良い点
- ...

### 改善提案
1. [高] 具体的な問題と改善案
2. [中] 具体的な問題と改善案
3. [低] 具体的な問題と改善案
```

## 注意事項

- レビュー対象がこのスキル自身の場合も、同じ基準で客観的に評価する
- スコアは相対評価ではなく、上記の絶対基準に基づく
- 改善提案は抽象的な助言ではなく、具体的なリライト案を含める
- ファイルの目的・対象読者（人間 vs エージェント vs 両方）を考慮して評価する
